# CLAUDE.md

このファイルは Claude Code が本リポジトリで作業する際の前提情報を提供する。Claude Code は起動時に本ファイルを必ず読み込み、以下のルールに従って動作する。

---

## プロジェクト概要

**ウルトラZAIMUくん LEO 版 PWA**。岡山県のスナック・小規模店舗向け会計＋勤怠＋月末経理ツール。3デバイス対応（スマホ／iPad／PC）。販売主体は株式会社ターゲット社。

- リポジトリ：`kana19/ultra-z-leo`
- 本番URL：`https://kana19.github.io/ultra-z-leo/`
- バックエンド：Google Apps Script + Googleスプレッドシート
- 詳細仕様：チャット側の Claude プロジェクト知識（システム仕様書.md・技術仕様書.md・3デバイス統合仕様.md・戦略思想メモ.md 等）に格納されている。Claude Code はこれらを参照する手段を持たないため、本ファイルに記載された範囲で動作する

## 最上位の判断軸（絶対遵守）

### 1. AI自動確定の禁止

ユーザー入力フローでは必ず「AI提案 → ユーザー確認 → ユーザーが確定ボタンを明示的にタップ」の3ステップを実装する。AI が確定まで自動で進める設計は禁止。レシートOCR・CSV科目分類・月末経理・全機能でこのルールが適用される。

### 2. 商売の都合優先

「システムのために商売を変えさせる」設計は禁止。深夜営業中・片手操作・騒音下での運用を前提とする（特に勤怠・売上入力UI）。複雑な技術判断より、現場で迷わず使える設計を優先する。

### 3. 物理分離（方式α）

顧客ごとに独立したリポジトリ・スプレッドシート・GASを持つ。1顧客の不具合が他顧客に波及しない構造を破壊する変更は禁止。

### 4. 業態テンプレート連動の実装方式

UI用語切替（「入店/出勤」「退店/退勤」等）は templateId 連動で動的に書き換える。HTML は固定の日本語表記のまま保持し、起動時に JS が書き換える方式。

- HTML 静的テキスト：`data-uilabel-key` 属性付与 → `applyUILabels()` が書き換え
- JS 動的生成テキスト：`deriveUILabels()` の戻り値を文字列に直接組み込み
- HTML `<title>`：body の `data-uilabel-doc-title-key` + `data-uilabel-doc-title-suffix` 属性
- 複合ラベル（例：「入店・退店修正」）：14キーを文字列連結して構築。新規キー追加は行わない

新規にラベルキーを追加する判断は本ファイル単独で行わず、必ずチャット側の指示書に基づく。

### 5. アプリ名表記の統一

正式名称：**ウルトラZAIMUくん**。短縮名：**ZAIMUくん**。「ウルトラ財務くん」は旧表記であり残すべきでない。サイドバーロゴ・JSコメント・CSSコメント等で統一する。

## やってはいけないこと（明示禁止リスト）

- `git push --force` / `git rebase` 等の履歴改変系操作（チャット側からの明示指示がある場合のみ実行）
- mainブランチ以外への push（プルリクエストは作成しない）
- ファイル削除（`git rm`）の単独判断（必ず指示書で明示されている場合のみ）
- npm / yarn 等の依存追加の単独判断（本プロジェクトはバニラJS構成・依存追加は事前合意必須）
- スマホ・iPad版コスト入力UIへの源泉徴収UI追加（戦略思想§3-5「業務×デバイス分担」違反）
- スマホ版の月末経理機能搭載（PC版のみに搭載される）
- AI自動確定フロー（前項参照）
- ブラウザストレージAPI（localStorage / sessionStorage）以外の永続化機構の独断追加

## コミット運用

- コミットメッセージは指示書で指定された文言を使用する
- 指定がない場合は派生論点番号 + 簡潔な日本語要約（例：`派生論点B: PC版業態切替対応`）
- Co-Authored-By trailer は付与してよい（事業運営者が許容済み）
- 1論点 = 1コミットを原則とする（複数論点をまとめない）

## チャット側 Claude との連携

本プロジェクトは「3本柱の会話スコープ」（構築系／資料作成系／登録手続き系）で運用される。Claude Code が受ける指示は構築系会話から発出される。

- 仕様の不明点・前提の曖昧さに遭遇した場合：**自己判断で実装せず、チャット側に戻して確認する**
- 戦略思想・販売価格・補助金登録方針・グレード差別化等の上位判断軸は**チャット側のプロジェクト知識が一次ソース**
- 本ファイルとプロジェクト知識が矛盾した場合：**プロジェクト知識を正とし、チャット側に齟齬を報告する**

## 動作確認

実装完了後の動作確認は、Claude Code 内で以下のいずれかで実施する：

- ローカルHTTPサーバ起動（`python3 -m http.server 8000` 等）
- 該当HTMLを開いてコンソールログを確認
- GAS 通信が必要なケースは事前にチャット側で「動作確認方法」が指示書に明記される

GitHub Pages反映後の本番動作確認は、ユーザー側がシークレットタブで実施する（Service Workerキャッシュ回避のため・チャット側で別途指示される）。
